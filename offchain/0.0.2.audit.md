## Critical — block publish

### C1. `fetch-submissions.ts` groups decode failures under a synthetic `unknown_*` proposal

**Location:** `src/helpers/fetch-submissions.ts:212, 273-277` · **Status:** CONFIRMED

When `parseCosponsorDatum()` fails, the submission is keyed under `unknown_${txId.slice(0,8)}` and grouped with placeholder `anchor: { url: "unknown", hash: "unknown" }` and `action: { kind: "Unknown" as any }`. Multiple failed submissions collide under the same synthetic key, mislabeling unrelated deposits as a single "proposal". Same family of bug as the SDK's previously-fixed "first UTxO anchor" fallback — both silently fabricate identity when decoding fails.

**Proposed action:**
1. Delete the synthetic-key path at line 212 and the placeholder grouping at lines 273-277.
2. On parse failure: `logger.warn("[fetch-submissions] datum decode failed for tx <hash>; skipping", { error })` and `continue` to the next UTxO.
3. Extend the return type so callers can opt into malformed submissions when they need to surface them:
   ```ts
   interface IFetchSubmissionsResult {
     proposals: IProposalSubmission[]
     malformed: { txHash: string; outputIndex: number; reason: string }[]
   }
   ```
4. Confirm no consumer of `fetchSubmissions` relies on `unknown_*` entries appearing in the result. Grep `cosponsor-ui` for "unknown_" / "kind === 'Unknown'" before deleting.

**Trade-off:** Returning `{proposals, malformed}` is a breaking change to the function signature, but the SDK is pre-1.0 — better to break now than after publish.

**Decision:**Approved

**Validated:** CONFIRMED. The working-tree file already returns `{validSubmissions, malformedSubmissions, totalStats}` (the "extend return type" step was done in a prior pass), but the synthetic `unknown_*` key path (and `null_datum_*`/`hash_datum_*`/`no_datum_*`/`error_*` siblings) plus the placeholder proposal `{anchor:"unknown", action:{kind:"Unknown"}}` were still live. **Done:** extracted a pure, provider-free `groupClassifiedSubmissions(IClassifiedSubmission[])` that routes every undecodable/unvalidatable UTxO to `malformedSubmissions` and skips it — decode failures can no longer fabricate a group, and two failures from the same tx no longer collide. `fetchAllSubmissions` now scans UTxOs into `IClassifiedSubmission[]` and delegates. Validation (unspent check) now only runs for decodable UTxOs. Tests: `tests/fetch-submissions-grouping.test.ts` (7 cases — valid grouping, same-hash collapse, After→Completed, decode-failure→malformed, same-tx no-collision, validation-failure→malformed, mixed-batch stats, empty input). Full suite 40/40. No consumers depend on `unknown_*` (grep of repo + UI clean). _Note: a pre-existing `error.message`-on-`unknown` tsc error in `validateDepositStructure` was left untouched (out of scope)._

### H1. Two divergent datum-parse paths with silent data loss

**Location:** `src/helpers/parseCosponsorDatum.ts:51-103` (uses `fromContractType`); `src/browser/fetchUserDeposits.ts:412-499, 693-783` (uses `typedActionFromSchema`); `fromContractType` itself at `src/validators/Types/GovernanceAction.ts:315-379` · **Status:** CONFIRMED

`fromContractType` hardcodes `beneficiaries: []` (line 341) and `addedMembers: []` (line 357) — TreasuryWithdrawal and ConstitutionalCommittee variants lose their actual data. The browser path's `typedActionFromSchema` does a proper PlutusMap walk via `credentialMapFromPlutusData` and returns real beneficiaries. Callers that pick the wrong path silently get empty arrays.

**Proposed action (Option A — preferred):** Retire the `[]` shortcut entirely.
1. Move `credentialMapFromPlutusData` (currently inside `fetchUserDeposits.ts`) up to `src/helpers/credentialMap.ts` so both decode paths can share it.
2. Rewrite `fromContractType`'s `TreasuryWithdrawal` and `ConstitutionalCommittee` cases to actually walk the PlutusMap, mirroring `typedActionFromSchema:730-757`.
3. Add a unit test for each that asserts a non-empty beneficiary set round-trips.

**Proposed action (Option B — bigger change, cleaner end-state):** Collapse to one path.
1. Have `parseCosponsorDatum` call `typedActionFromSchema` directly. Delete `fromContractType` entirely.
2. Audit every `fromContractType` caller and migrate them.

**Trade-off:** Option A is a smaller diff and lower-risk. Option B removes the second path entirely so this class of bug can't return. Recommendation: A now (unblocks publish), B in a follow-up commit.

**Decision:**Validate before addressing, but option A sounds sensible

**Validated:** CONFIRMED, and traced to the exact impact. `fromContractType` returns `beneficiaries: []` / `membersToAdd: []`. The data flow: `parseCosponsorDatum` sets `proposal.action = fromContractType(...)` (lossy) but ALSO preserves `rawCosponsoredProposal` (exact bytes). The **only** caller that re-hashes from the typed action is `fetch-submissions.getProposalHash` (`Cosponsor.new(...).gAda()` → `ToContractType(action)` → serialize+hash) — so it computed a **wrong gADA hash** for TreasuryWithdrawal/ConstitutionalCommittee. `depositIndexer` already hashes the raw and was fine. On-chain confirmation: the Aiken validator (`lib/calculation/conversion.ak:143-148`, `cosponsor.ak:206`) serialises the *whole* procedure opaquely and checks `proposal_procedure_hash == token_name` — it never inspects action contents, so only byte-exact CBOR matters. **Decision (with user): minimal fix, not Option A.** Made `getProposalHash` serialize `rawCosponsoredProposal` directly (matching `depositIndexer`), dropping the lossy `Cosponsor.new(...).gAda()` rebuild. `fromContractType` stays lossy-for-display (as its docstring already documents). Tests: `tests/proposal-hash-from-raw.test.ts` (4 cases, TreasuryWithdrawal with 2 real beneficiaries: datum round-trips; raw-hash == build-time token name; typed action drops beneficiaries; **rebuild-from-typed gives a different/wrong hash** — proving the bug). `bun run build` ✓, suite 60/60 ✓, only pre-existing tsc error remains in this file.

### H2. `typedActionFromSchema` silently discards `NewConstitution` constitution data

**Location:** `src/browser/fetchUserDeposits.ts:778-779`; matching builder `src/validators/Types/GovernanceAction.ts:660-689` · **Status:** CONFIRMED

Both the reader and the builder ignore `constitutionHash` / `constitutionUrl` and always encode `guardrails: None`. The round-trip hash check at `fetchUserDeposits.ts:875` passes only because both sides forget the same data. The next maintainer who "fixes" one side silently breaks proposal-hash stability for NewConstitution proposals.

**Proposed action (Option A — recommended):** Round-trip the constitution fields properly.
1. Update `typedActionFromSchema` (lines 778-779) to read the constitution fields from the parsed PlutusData.
2. Update `buildNewConstitutionAsPlutusData` (`GovernanceAction.ts:660-689`) to encode them.
3. Add `guardrails` handling if needed (currently always `None`).
4. Add a round-trip test for NewConstitution with realistic hash/url values.
5. Verify against the Aiken validator: does it accept these fields when non-empty? If the on-chain script ignores them, our datum becomes informational-only but still needs to be hash-stable.

**Proposed action (Option B — document the invariant if A is too much for this version):** Add a guarded comment block above both sites:
```ts
// INVARIANT: NewConstitution constitution metadata is intentionally not
// round-tripped. Both this reader and `buildNewConstitutionAsPlutusData`
// MUST stay in lockstep — fixing one without the other breaks the
// proposal-hash check at fetchUserDeposits.ts:875.
```

**Open question:** Is the on-chain validator currently tolerant of non-empty constitution fields? Need to check the Aiken source before picking Option A.

**Decision:**A, and I would argue if possible we should add test suite cases for all proposal types and properly structured/malformed data to ensure no undefined/unexpected behavior happens.

**Validated:** REFRAMED — the audit's premise was wrong, and the user pushed to check we weren't dropping data we shouldn't. The on-chain `Constitution` type (aiken-lang-stdlib `cardano/governance.ak:96`) is **`Constitution { guardrails: Option<ScriptHash> }`** — it has **no document anchor at all**. So `constitutionHash`/`constitutionUrl` on `INewConstitution` map to nothing on-chain (vestigial), and the audit's "round-trip the constitution fields" (Option A) was impossible. The validator also serialises the procedure opaquely (`conversion.ak:143-148`) — only byte-exact CBOR matters. The *real* gap: the one field that DOES have an on-chain slot — `guardrails` — was hardcoded to `None` and unsettable.

**Decision (user, aiming for production-ready + complete): Option 4 — full realign.**
1. Added optional `guardrails?: string` to `INewConstitution`; **deprecated** `constitutionHash`/`constitutionUrl` (made optional, JSDoc `@deprecated`).
2. Wired `guardrails` through **both** build paths — `ToContractType` (schema/serialize) and `buildNewConstitutionAsPlutusData` (raw PlutusData) — `None` encodes byte-identically to before (so existing proposals stay hash-stable), `Some(hash)` = `Constr(0,[ByteArray])`.
3. Both readers (`fromContractType`, `typedActionFromSchema`) now surface `guardrails` and drop the vestigial fields.
4. Tests: `tests/new-constitution-guardrails.test.ts` (Some round-trips, None→undefined, rebuild hash-lossless, Some≠None token name) + `schema-probe.test.ts` updated (dropped vestigial fields; added Some schema==builder equivalence + None≠Some). `bun run build` ✓, suite 66/66 ✓, no new tsc errors.

**Flagged (out of offchain scope, your call):** this protocol's on-chain `Constitution` lacks a CIP-1694 document anchor entirely. If full Cardano-governance fidelity is a goal, that's an on-chain Aiken change + redeploy + re-audit — not an SDK fix.

### H3. `datum.kind() === 1` inline-datum check inconsistent across three files

**Location:** `src/helpers/depositIndexer.ts:122-123`, `src/helpers/fetch-submissions.ts:177-179`, `src/browser/fetchUserDeposits.ts:1064-1065` · **Status:** CONFIRMED

Two files use `if (datum.kind() === 1) datum.asInlineData()`. The third uses `datum.asInlineData?.() ?? datum` — optional chain plus raw-datum fallback. A non-inline datum at the third site quietly coerces to something unintended.

**Proposed action:**
1. Add `src/helpers/datumUtils.ts`:
   ```ts
   export const extractInlineDatum = (datum: Core.Datum | null | undefined): Core.PlutusData | null => {
     if (!datum) return null
     if (typeof datum.kind !== "function") return null  // some Datum impls expose this differently
     return datum.kind() === 1 ? datum.asInlineData() ?? null : null
   }
   ```
2. Replace all three call sites with `extractInlineDatum(...)`.
3. Pick a single behavior for "datum hash only, no inline": currently the third site coerces to the raw object; the other two return `undefined`. The helper returning `null` standardizes this — verify no caller relied on the coercion.

**Decision:**Reusable code is good, means bugs are fixed at once, but should be tested properly to ensure we're not introducing breaking changes to all call sites

**Validated:** CONFIRMED. The divergent/risky site is `fetchUserDeposits.ts` (`datum.asInlineData?.() ?? datum` — no `kind()` check; a hash-only datum coerced the raw datum object into `PlutusData`). **Done:** added `src/helpers/datumUtils.ts` → `extractInlineDatum(datum): Core.PlutusData | null`. Adopted it at all three sites, preserving behavior:
- `depositIndexer` (site 1) and `fetchUserDeposits` (site 3, the buggy one) now use it directly; site 3 guards `null` before the extractors so a hash-only datum yields the same `decodingFailed:true, hasDatum:true` result as the old coercion (no behavior change, just explicit).
- `fetch-submissions` (site 2) uses the helper for *extraction* but keeps its distinct malformed reasons (`no datum` / `hash-only` / `inline null`) because C1's `malformed` reporting surfaces them — replacing with the generic helper would have lost that granularity.

Cross-package gotcha (validated via tsc): `Core.Datum` is methodless at the type level (the `@cardano-sdk/core` vs `@blaze-cardano/core` divergence the old code commented on), so the helper takes a **structural** `{ kind?(); asInlineData?() }` param — every call site passes its own `Datum` with no cast. Tests: `tests/datum-utils.test.ts` (6 cases: inline→data, hash-only→null, null/undefined→null, no-kind→null, asInlineData→undefined→null, no-asInlineData→null). `bun run build` ✓, suite 56/56 ✓, no new tsc errors. (Prettier flags these files for pre-existing CRLF line endings + pre-existing wrapping — not from these edits; not reformatted, to avoid a repo-wide churn.)

## Methodology

Three independent passes scoped non-overlappingly (browser flows / validator+datum / Node+cross-cutting). Each produced 10-15 findings against the same severity scale. Two validators then spot-checked the most consequential claims by reading actual code, returning CONFIRMED / PARTIALLY CONFIRMED / REFUTED. Refuted claims are listed above for transparency; the count dropped from ~40 raw findings to ~30 after dedup and validation.
